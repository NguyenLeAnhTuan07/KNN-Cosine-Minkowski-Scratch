# KNN Cosine & Minkowski Framework

A from-scratch implementation of the **K-Nearest Neighbors (KNN)** classifier supporting two distinct geometric approaches — Cosine Similarity and Minkowski Distance — with professional-grade preprocessing pipelines. No high-level machine learning libraries are used in the core algorithm.

**Author:** Nguyen Le Anh Tuan, Nguyen Duc Huy

---

## Overview

This project goes beyond a textbook KNN implementation by incorporating robust data-cleaning techniques and automated hyperparameter tuning. The result is a classifier that handles real-world data — including outliers, skewed distributions, and varying feature scales — without requiring external ML frameworks.

---

## Two Geometric Approaches

### 1. Cosine Similarity

This approach measures the **angle** between feature vectors rather than the distance between them, making it well-suited for cases where the pattern of the data matters more than its absolute magnitude.

**Preprocessing:** L2 Normalization — maps all samples onto a unit sphere so that only the direction of each vector is compared.

**Best used when:**
- Analyzing behavioral or usage patterns (e.g., telecom customer segmentation)
- Feature scales vary significantly across columns (e.g., income vs. age)
- The relative proportion of features is more meaningful than raw values

### 2. Minkowski Distance (Euclidean & Manhattan) with Robust Scaling

This approach uses the Minkowski metric with configurable `p`, allowing a switch between **Euclidean** (`p=2`) and **Manhattan** (`p=1`) distance. A two-stage preprocessing pipeline is applied to ensure robustness on noisy, real-world data.

**Preprocessing pipeline:**
- **Clipping** — removes extreme outliers by capping values at the 1st and 99th percentiles
- **Robust Scaling** — normalizes features using the median and interquartile range (IQR), resistant to skewed distributions

**Best used when:**
- Working with physical or clinical measurements (e.g., biomechanical orthopedic features)
- The dataset contains noise or significant outliers
- The physical distance between data points carries direct interpretive meaning

## How to Use

### 1. Prepare your dataset

- Place your dataset in CSV format into the `data/` directory and rename it `data.csv`
- Update `feature_names.txt` to match your column names, one per line
- The **last name** in the list must be the target label (the column to predict)

### 2. Encode categorical labels (if needed)

If your dataset contains string labels such as `"Normal"` or `"Abnormal"`, the mapping is handled automatically by `encoding/encoding.py` before any scaling is applied. No manual step is required.

### 3. Add samples to predict

Place the new samples you want to classify in `dudoan.csv`, using the same columns as `data.csv` but **without** the target label column.

## Installation & Execution

**Create a virtual environment (recommended)**

```bash
python -m venv venv

# Linux / macOS
source venv/bin/activate

# Windows
venv\Scripts\activate
```

**Install dependencies**

```bash
pip install numpy pandas
```

**Find the optimal K via cross-validation**

```bash
python validation.py
```

Runs K-Fold Cross Validation (5-Fold by default) across all odd values of K up to √N, printing the accuracy at each K and returning the best value.

- If `scale/scaler_config.pkl` does not yet exist, `validation.py` automatically calls `scale.py` to fit and save the scaler before running validation
- If the scaler already exists, `validation.py` loads the saved params from `scaler_config.pkl` to ensure consistency with `predict.py`

**Run full evaluation**

```bash
python evaluate.py
```

Runs a complete K-Fold evaluation of the KNN model on the scaled training data (`scale/data_scaled.csv`). If the file does not yet exist, `evaluate.py` triggers preprocessing automatically.

At startup, the script prompts for:
- The aggregation method for **multi-class datasets** (Accuracy / Macro-Average / Weighted-Average) — this prompt is skipped automatically for binary classification
- The value of K and the distance metric (p=1 Manhattan or p=2 Euclidean)

It then reports for each fold and in aggregate:
- **Accuracy, Precision, Recall, F1** — per fold and overall mean
- **AUC** — computed via vote-ratio scores (One-vs-Rest for multi-class) and the trapezoid method, entirely from scratch
- **Confusion matrix** — full matrix across all folds combined, displayed as a labeled grid
- **Per-class metrics** — TP, TN, FP, FN, Precision, Recall, F1 and Support for each label
- **CV Score** — mean and standard deviation for whichever metric you select

Detailed per-sample predictions are saved to `predict/knn_eval_details.csv`.

**Run predictions**

```bash
python predict.py
```

Classifies each row in `dudoan.csv` and prints the predicted class for each sample.

- If `scale/scaler_config.pkl` does not yet exist, `predict.py` fits and saves the scaler on the training data first, and exports `scale/data_scaled.csv`
- After scaling `dudoan.csv`, a corresponding `scale/dudoan_scaled.csv` is also saved automatically

## Multi-Class Metric Aggregation

When the dataset has more than two labels, `evaluate.py` asks which aggregation method to use for Precision, Recall, and F1. The three options correspond to the following formulas:

**Accuracy** — the proportion of all samples that were classified correctly:

```
Accuracy = (sum of diagonal of confusion matrix) / total samples
```

**Macro-Average** — computes each metric per class, then takes the simple mean, treating all classes equally regardless of size:

```
Macro_Precision = (Precision_1 + Precision_2 + ... + Precision_n) / n
```

**Weighted-Average** — computes each metric per class, then takes the weighted mean using each class's sample count (`s_i`) as the weight, giving more influence to larger classes:

```
Weighted_Precision = (s_1 * Precision_1 + s_2 * Precision_2 + ...) / (s_1 + s_2 + ...)
```

For binary classification these prompts are skipped and metrics are computed directly from the single positive class.

## Key Design Decisions

**Why no sklearn for the core algorithm?**
The KNN logic — distance calculation, neighbor selection, and majority voting — is implemented entirely with NumPy. This makes every step transparent and easy to trace, which is the primary goal of this project.

**Why is `scale.py` a standalone module at the project root?**
The scaling algorithm (Clipping + Robust Scaling) is a pure mathematical transformation with no dependency on file I/O or project-specific paths. Isolating it in `scale.py` keeps the math clean and reusable — `preprocessing.py` handles all file reading, writing, and path logic and simply calls into `scale.py` for the computation. This separation also means `validation.py` and `predict.py` can check `is_fitted()` and call `fit_scaler` or `transform_scaler` independently, without going through the full preprocessing pipeline every time.

**Why does `scale.py` also export CSV files?**
Each time a scaling operation is performed — whether on training data or on `dudoan.csv` — a human-readable CSV of the scaled result is saved alongside the binary `scaler_config.pkl`. This makes it easy to inspect what the model actually sees as input, verify the scaling visually, and feed the scaled files directly into `evaluate.py` without re-running the pipeline.

**Why two separate preprocessing pipelines?**
Cosine and Minkowski distance measure fundamentally different things. Cosine cares only about vector direction, so L2 Normalization is the correct and sufficient preprocessing step. Minkowski cares about magnitude, so outlier removal and robust scaling are necessary first. Mixing the pipelines would undermine the geometric guarantees of each metric.

**How does `evaluate.py` compute AUC without sklearn?**
For binary classification, `evaluate.py` uses the KNN vote ratio for the positive class as a confidence score. For multi-class, it applies a One-vs-Rest strategy — computing a separate AUC for each label and averaging the results. In both cases the area under the ROC curve is calculated from scratch using the trapezoid method over the score-sorted sequence.

**Why is multi-class aggregation optional and prompted at runtime?**
Accuracy, Macro-Average, and Weighted-Average answer different questions and are appropriate in different situations. Rather than hardcoding one approach, `evaluate.py` asks the user to choose — but only when there are more than two labels, since the distinction is only meaningful in the multi-class case.
